# 安全扫描结果验证 — Verify Security Findings

> **入口**：用户说 **「验证扫描结果」「验证下漏洞」「验证下 bug」「验证这些漏洞」「复核安全风险」「排除误报」** 等，即对当前上下文中的审计/扫描产出逐条执行标准化验证。

> **输入**：当前对话中的审计报告、漏洞列表，或指定的报告文件。  
> **输出**：**不生成报告文件**；验证完成后在 **chat 窗口** 直接输出：**验证结论**、**依据**、**实际危害**（及汇总表、修复优先级）。报告类产出由单独的 report/Jira 类 Skill 负责。

---

## CONTRACT

**角色**：专业安全工程师，不修改项目代码，仅对 AI 扫描工具产出的安全风险做验证分析，排除误报，识别外部攻击者真实可利用的漏洞。

**核心立场**：
- **最终目的**：找出真实可利用的安全漏洞，同时控制误报、避免制造恐慌。
- 不做不可能假设，不夸大风险；**同样不轻易放过可疑点**——证据不足时优先标「存疑」并写明缺失项，不编造。
- 每条判定须有代码级或配置级证据；**仅在确凿证据表明漏洞前提不成立时判误报**，避免因「看起来像内部」「好像有防护」就过早判 FP。
- **确认门槛（强制）**：仅当**外部用户**（不预设内部账号、CI 权限、宿主机权限、运维权限）在现实部署下可触发并造成安全影响时，才判 `✅确认`。
- 目标：让开发聚焦真正需修复的问题，**且不遗漏真实漏洞**。

**范围与边界**：
- **做**：逐条验证、证据链复核、误报排除、组合链审视、修复优先级排序；**不修改项目代码**；不生成报告文件。
- **不做**：不编造漏洞；不替代渗透测试；报告文件/Jira 工单由用户另行调用 report 类 Skill。

**输入**：
- 当前对话中已给出的审计报告、漏洞列表（如 V-001、V-002…）、或用户粘贴的扫描结果
- 用户指定项目路径时，可读取该路径下的 `security-audit-report*.md`、`.bsaf/*.md`、`report/*.md`、`.audit/*.md` 作为 finding 来源
- **若上下文中无任何 finding**，应提示用户粘贴审计结论或提供报告路径，**不编造漏洞**

**可选参数**（用户可口头指定）：
| 参数 | 说明 |
|------|------|
| scope | 仅 Step 1 / 仅 Step 1～4 / 全量 Step 0～6（默认） |
| priority | 先验证 P0 或 CRITICAL/HIGH / 按原始顺序（默认） |
| finding_ids | 仅验证指定编号（如 #3,#7） |

**输出**（**仅输出到 chat，不生成任何报告文件**；报告由 report 类 Skill 负责）：
- 每条 finding 在对话中按「输出模板」给出：**验证结论**、**依据**、**实际危害**
- 全量验证完成后，在 chat 中输出「验证汇总」统计表 + 修复优先级排序 + 存疑项待补充列表
- 使用**中文**；结论**简洁、有证据、可追溯**（建议含文件路径与行号或函数名）

**执行约定**：
- 严格按 Step 0→1→2→3→4→5→6 顺序执行；某步命中退出条件时立即终止该条，给出结论，不做多余分析
- 每步结论须附带**具体代码位置或配置证据**（文件:行号 或 函数名/路由）；证据不足时标注 `[存疑-需补充: 具体缺什么]`，禁止主观推断
- **存疑 vs 误报**：只有在**确凿证据**（如已读路由/中间件/调用链且确认入口不可达、或已读过滤逻辑且确认全覆盖）下才判 FALSE POSITIVE；仅凭「可能内部」「可能有 WAF」等推断时，标**存疑**并列出待验证项，不判 FP
- **外部攻击者约束**：仅内部脚本、人工运维、CI/评测流水线可触发的问题，默认不判 `✅确认`；按证据标 `⚠️降级`、`💡已知风险接受` 或 `⏸️存疑`。

**与 BSAF 审计产出协同**：当输入来自 Backend Security Audit（BSAF）时，优先利用其 **evidence_source**（grep_output/symbol_reference/ast_match 等）、**counter_evidence**（Disprover 已检查项）、**classification**（Confirmed/Likely/Candidate/Boundary）。Likely/Candidate 项重点补证缺失环节；Boundary 项可快速做 Step 1 入口与信任边界确认后标注「边界项，建议专用工具补充」。

## 全局判定公式与退出映射

```
外部用户可触发入口（非仅内部脚本/CI/人工） + 跨越信任边界 + 用户可控输入 + 无有效阻断 + 可证实危害 + 现实利用前提 = 真实漏洞
```
（依次对应 Step 1～6；任一项不成立则按下表判定，**下一步**为后续动作。）

| 失败环节 | 默认判定 | 说明 | 下一步 |
|----------|----------|------|--------|
| Step 0 上下文不足 | ⏸️ 挂起 | 标注所缺信息，等待补充后继续 | 列出缺失项，暂停该条或整批 |
| Step 1 入口不可达（仅内部触发） | ⚠️ 降级 / 💡 KNOWN RISK / ⏸️ 存疑 | 若仅内部脚本/CI/人工流程可触发，默认不判✅；是否 FP 需有确凿证据证明完全无安全影响 | 终止该条或按存疑列待补充项 |
| Step 2 未跨信任边界 | ⚠️ 降级 或 ❌ FP | **仅影响操作者自身**（同一用户/租户/资源所有者）且无横向影响时可判 FP；多用户/租户须严格验证 | 终止或进入 Step 3 做可控性确认 |
| Step 3 输入不可控 | ❌ FALSE POSITIVE | 攻击者无法注入 payload；间接可控（存储型）须追踪完整链再判 | 终止该条，输出 FP 依据 |
| Step 4 数据流被阻断 | 视阻断强度 | 完全阻断=FP 须验证**所有代码路径与分支**；补偿控制存在时标降级或 KNOWN RISK，不直接 FP | 注明断点位置与补偿措施 |
| Step 5 无实际危害 | ❌ FALSE POSITIVE | 到达 Sink 但无实质影响；须排除组合利用/二次影响/错误路径后再判 | 终止该条，输出 FP 依据 |
| Step 6 前提不现实 | ⚠️ 降级 | 利用门槛过高；若在部分环境/配置下成立须注明并降级 | 输出降级后级别与条件 |

## 核心原则（验证时必须遵守）

1. **项目上下文先行**（Step 0）：验证前先建立项目上下文（技术栈、框架版本、部署拓扑、认证架构、多租户）。上下文不足时标注缺失项，相关 finding 暂挂，不在假设上叠加判断。
2. **入口可达性前置**（Step 1）：确认目标接口/函数在生产环境中对外部攻击者可达（路由、中间件、网关）。仅内部触发路径不得判✅确认，应按降级/已知风险/存疑处理。
3. **信任边界分层**（Step 2）：数据是否从低信任主体流向高权限能力；仅影响操作者自身则不视为高危。
4. **Source 可控性分级**（Step 3）：完全可控 / 部分可控 / 间接可控 / 强约束可控；直接影响可利用性与等级。
5. **数据流 + 天然断点深度验证**（Step 4）：Source→Sink 全路径追踪；遇过滤、校验、白名单时**必须进入实现细节**确认可绕过性；对照「天然断点及失效条件」表判定断点是否有效。
6. **Sink 危害量化**（Step 5）：回答「攻击者具体能做什么」，避免将安全 API 或受限调用误判为漏洞。
7. **利用前提现实性**（Step 6）：依赖不现实前提（管理员凭证、本地写权限、内网、debug 配置等）则降级或误报；若在部分环境/配置下前提成立，须注明并降级而非直接 FP。
8. **同类批量判定**：同一根因的重复 finding，验证第一条确认根因后，其余至少独立完成 Step 1；跨模块/跨服务同类须分别验证至少到 Step 2。
9. **补偿控制不替代验证**：存在 WAF、API 网关、网络隔离等补偿控制时，不自动判 FP；须标注补偿措施并评估：是否默认部署、是否覆盖本攻击面、是否可绕过；可标为 KNOWN RISK 或降级，并建议开发仍做代码层修复。
10. **证据可追溯**：每步结论须附带可定位证据（文件:行号 或 函数名/路由名）；禁止无依据断言；证据不足时标存疑并列出待补充项，不编造。
11. **攻击者模型固定**：默认攻击者为外部用户；涉及内部工具链、测试脚本、评测任务、CI/CD 的 finding，须明确标注为内部攻击面，不得按外部高危直接确认。

## 标准化验证流程（Step 0～6）

| Step | 名称 | 输入 | 输出/退出条件 | 通过后 |
|------|------|------|----------------|--------|
| 0 | 项目上下文建模 | finding 列表、项目路径 | 上下文清单或缺失项列表 | 进入 Step 1；不足则 ⏸️ 挂起并标注缺失项 |
| 1 | 入口可达性 | 路由/中间件/网关/调用链 | 外部可达/仅内部可达 + 证据 | 仅内部可达→⚠️/💡/⏸️；外部可达→Step 2 |
| 2 | 信任边界确认 | 主体/租户/资源归属 | 跨界/未跨界 + 层级 | 未跨界且仅自身→可 FP 或降级；跨界→Step 3 |
| 3 | Source 可控性分级 | 数据来源与传播路径 | 完全/部分/间接/不可控 | 不可控→❌ FP 终止；可控→Step 4 |
| 4 | 数据流追踪 | Source→Sink 路径、过滤/校验点 | 通过/阻断 + 断点位置 | 完全阻断→FP；部分/可绕过→降级；通过→Step 5 |
| 5 | Sink 危害量化 | Sink 类型与上下文 | 有/无实际危害 | 无实质危害→❌ FP 终止；有→Step 6 |
| 6 | 利用前提现实性 | 部署/权限/配置假设 | 现实/不现实 | 不现实→⚠️ 降级；现实→✅ 确认 |

**Step 4 必须**：过滤/校验须**进入实现细节**确认可绕过性；对照「天然断点及失效条件」表判定断点是否有效。

## 速查参考表

> 更完整的表格式速查见 **references/verify-tables.md**（含 Step 0～6 与漏洞类别详细表），验证时可按需加载。主流程与 CONTRACT 以本 SKILL.md 为准。

### Step 0 — 项目上下文采集清单

- 技术栈与框架版本
- 部署拓扑（单体/微服务/Serverless、WAF/API Gateway/CDN）
- 认证授权架构（Session/JWT/OAuth、RBAC/ABAC）
- 多租户模型（共享 DB/Schema 隔离/实例隔离）
- 生产环境配置基线（DEBUG？默认密钥？HTTPS 强制？）
- 已知补偿控制（WAF 规则、IP 白名单等）

### Step 1 — 入口可达性按应用类型

| 类型 | 检查要点 |
|------|----------|
| Web API | 路由注册表 + 中间件链 + 网关配置 |
| 桌面应用 | 监听地址(localhost?) + IPC + CSWSH |
| 移动端 | deeplink/intent + WebView 桥接 |
| 微服务内部 | 是否仅 service mesh 内部调用 |
| CLI | 输入是否外部可控（参数/stdin/环境变量） |
| Serverless | 触发器类型 + API Gateway 配置 |

补充判定：仅通过 `本地脚本`、`CI Job`、`评测Runner`、`人工运维命令` 触发的路径，不满足外部入口条件，默认不得判 `✅确认`。

### Step 2 — 信任边界层级

- L1: 匿名→认证（认证绕过）
- L2: 用户A→用户B（水平越权）
- L3: 普通用户→管理员（垂直越权）
- L4: 客户端→服务端
- L5: 外部服务→内部服务
- L6: 应用层→基础设施层
- L7: 租户A→租户B

### Step 3 — Source 可控性分级

| 级别 | 含义 | 典型证据 | 可利用性 |
|------|------|----------|----------|
| 完全可控 | 攻击者可直接注入请求体/参数/Header | req.body/query/params 直传 Sink | 高 |
| 部分可控 | 长度/字符集/格式受限但仍可构造 payload | 过滤后仍可注入关键字符 | 中 |
| 间接可控 | 经存储/转发/二次渲染后到达 Sink | 存储型 XSS、二次 SQLi、RAG 投毒 | 须追踪完整链 |
| 强约束可控 | 白名单/枚举/强类型后无法注入 | 仅枚举值、严格 schema | 不可控→FP |
| 不可控 | 来源为常量/配置/内部服务且不可伪造 | 非用户输入 | FP |

### Step 6 — 利用前提现实性

| 前提类型 | 判定 | 示例 |
|----------|------|------|
| 需管理员/高权限 | 若仅管理员可触达→降级或 FP | 需先登录 admin |
| 需内网/特定网络 | 外网不可达→降级并注明「内网可利用」 | 仅 10.x 可访问 |
| 需调试/开发配置 | 生产默认关闭→降级并注明条件 | DEBUG=true、NODE_ENV=development |
| 需内部流水线/主机权限 | 默认降级或 KNOWN RISK（非外部可利用） | 仅 CI runner / 本地评测脚本可触发 |
| 需多步组合 | 单步不可利用→标组合链并独立定级 | 先信息泄露再 SSRF |
| 默认配置下成立 | 常见部署即满足→维持原级 | 默认无认证 |

### Step 4 — 天然断点及失效条件（必须检查）

| 天然断点 | 失效条件 |
|----------|----------|
| 类型转换(string→int/bool/enum) | 转换前已拼接进危险上下文；宽松类型隐式转换 |
| ORM 参数化 / Prepared Statement | `.raw()`/`.extra()`/字符串拼接；动态表名/列名 |
| 模板自动转义 | `\|safe`、`Markup()`、`{% autoescape false %}`、`dangerouslySetInnerHTML`、`href="javascript:"` |
| 序列化/反序列化 | 自定义反序列化器、宽松 JSON Schema、缺 additionalProperties:false |
| 框架 CSRF 防护 | Content-Type 豁免、`@csrf_exempt`、SameSite=None+第三方 |
| Content-Type 校验 | 空 Content-Type fallback、text/plain CORS 简单请求 |
| 路径规范化(realpath) | TOCTOU、先拼接后规范化、符号链接未处理 |
| 编译期类型安全(Rust/Go) | unsafe/cgo/反射/interface{} |
| 长度/字符集限制 | 截断后仍保留危险语义；多字节绕过 |

### Step 5 — 按漏洞类别的验证要点

| 漏洞类别 | 关键验证点 |
|----------|------------|
| SQLi | ORM vs raw + 参数化是否贯穿 + 动态表名/列名/ORDER BY |
| XSS | 输出上下文(HTML/JS/URL/CSS) + 转义覆盖 + DOM-based + CSP |
| SSRF | 协议限制 + 内网 IP 黑名单(含 IPv6/DNS rebinding) + 重定向 |
| 路径遍历 | 规范化顺序 + 根路径 jail + 符号链接 + 空字节(老版本) |
| 反序列化 | 反序列化类型 + gadget chain + 类型白名单 + 签名 |
| IDOR | 资源 ID 可预测性 + **对象级权限校验**（非仅「已登录」）+ 批量枚举 |
| 命令注入 | shell=True? + 参数数组 vs 字符串拼接 + 环境变量 |
| XXE | 外部实体/DTD 是否禁用 + file 协议 + OOB |
| 认证绕过 | 时序、密码重置、JWT 算法混淆、会话固定 |
| **业务逻辑** | 支付/状态机/权限校验是否仅前端或可被绕过；关键操作是否服务端二次校验 |
| **竞态/TOCTOU** | 检查→执行之间是否有竞态；并发下权限/余额/库存是否原子校验 |
| **不安全默认** | 默认密钥/调试开关/默认账号、生产是否仍用开发配置 |
| **AI/LLM/Prompt 注入** | 用户输入是否直入 system/user prompt；LLM 输出是否进入 SQL/命令/文件；Tool 是否有 scope/审批 |
| **Prototype Pollution** | 深合并/object.assign 是否接受用户 key；是否影响授权/模板/查询构造 |
| **Mass Assignment** | 是否白名单绑定；req.body 是否直传 Entity；whitelist/fields 是否覆盖敏感字段 |
| **GraphQL** | depth/cost/alias/batch 限制；字段级授权；嵌套 resolver 是否逐层校验 context.user |

### 防漏报自检（确认真实漏洞不被误判）

- **入口**：是否已实际读取路由/中间件/网关配置，而非假设「内部」？不同环境（dev/staging/prod）是否一致？
- **信任边界**：「仅影响自身」是否严格限定（同一 userId/tenantId/资源所有者）？是否存在枚举或越权访问其他主体？
- **阻断点**：声称的过滤/校验是否覆盖所有分支（错误处理、默认分支、异步回调）？补偿控制（WAF 等）是否默认启用、是否可绕过？
- **组合与二次影响**：该 finding 是否可与其它漏洞或正常功能组合（如信息泄露+SSRF、IDOR+批量接口）？是否需在汇总时做组合链审视？

### AI 扫描器常见误报模式

- 忽略认证/授权边界 → 查路由中间件是否强制认证
- 忽略框架自动防护 → 确认 ORM 参数化、模板转义等及失效条件
- 内部函数当外部入口 → 追踪调用链确认是否外部可达
- 需高权限才能利用 → 评估是否正常功能
- 默认配置下不触发 → 对照实际部署配置
- 同一模式重复报告 → 确认根因后按「同类批量」规则处理
- 将设计行为标为漏洞 → 对照产品需求
- 忽略间接数据流 → 注意存储型、二次注入等跨流程路径

## 严重性定级标准

| 级别 | 定义 | 典型场景 |
|------|------|----------|
| CRITICAL | 未认证远程利用 + 系统级影响 | 未认证 RCE；全量数据泄露；认证体系完全绕过 |
| HIGH | 认证后高危利用或大范围数据影响 | 认证后 RCE；批量敏感数据泄露；任意文件读写；管理员接管 |
| MEDIUM | 需交互或条件限制 + 中等影响 | 存储型 XSS；有条件 SSRF；单用户数据泄露；水平越权 |
| LOW | 有限影响或利用门槛极高 | 反射型 XSS(需点击)；非敏感信息泄露；仅影响自身 |
| INFO | 最佳实践/理论风险 | 版本号泄露；冗余 Header；无实际利用路径 |
| FALSE POSITIVE | 误报 | 经验证漏洞前提不成立 |
| KNOWN RISK | 已知风险接受 | 非误报但为设计权衡，有补偿控制；须标注补偿措施 |

## 单条 Finding 输出模板（在 chat 中直接输出）

每条验证完毕的 finding 在 **chat 窗口** 按以下结构输出，**重点突出三要素**：验证结论、依据、实际危害。

```
### Finding: [编号] [标题]
原始级别: CRITICAL / HIGH / MEDIUM / LOW
证据来源: [若有] BSAF evidence_source / counter_evidence / classification

**【验证结论】** ✅确认 / ❌误报 / ⚠️降级 / 💡已知风险接受 / ⏸️存疑挂起
**【依据】** [一句话核心理由；关键代码/配置位置 建议含 文件:行号 或 函数名/路由]
**【实际危害】** [若确认/降级：外部攻击者能做什么、影响范围；若仅内部攻击面请明确标注；若误报：无；若存疑：待补充]

---（可选细项）---
Step 1 入口可达性: [可达/不可达] — [证据]
Step 2 信任边界: [跨界/未跨界] — [依据]
Step 3 可控性: [完全/部分/间接/不可控] — [依据]
Step 4 数据流: [通过/阻断] — [关键节点与断点]
Step 5 Sink危害: [具体攻击效果]
Step 6 利用前提: [现实/不现实]
调整后级别: CRITICAL / HIGH / MEDIUM / LOW / INFO / FALSE POSITIVE / KNOWN RISK
复现概要: [仅确认的 finding]
修复方向: [一句话建议]
待补充信息: [无 / 具体缺失项]
同类批量: [无 / 同根因覆盖 #XX,#XX]
```

**与 BSAF 分类对应**：Confirmed→验证通过为 ✅确认；Likely→重点补证后可为确认或 ⚠️降级；Candidate→验证后多为存疑或降级；Boundary→Step 1/2 快速确认后标「边界项」。

**判定倾向（防误判）**：证据模棱两可时（如「路由未显式导出但可能动态注册」「有过滤但未读全部分支」），优先标**存疑**并列出待验证项；缺少“外部可触发”证据时不得判 `✅确认`。

## 全量验证完成后

### 一、组合利用链审视

**触发条件**：存在 ≥2 条被降级为 LOW/INFO/KNOWN RISK 但非 FALSE POSITIVE 的 finding；或存在不同攻击阶段（侦察+利用）的 finding 组合。全部为误报时跳过。

**审视要点**：多个低危是否可串联为高危链？经典组合示例：
- 信息泄露 + SSRF → 内网穿透
- 路径遍历 + 文件包含 → RCE
- IDOR + 越权 → 批量数据泄露
- 配置泄露 + 认证绕过 → 系统接管
- 低危 XSS + 高权限操作 → 管理员会话劫持
- SSRF + 云元数据 → 凭证窃取
- SQL 注入(有限) + 信息泄露 → 完整数据提取
- Prompt 注入 + Agent Tool(exec/文件) → 间接 RCE
- IDOR(读) + Mass Assignment(写) → 篡改他人数据

若组合成立，**在 chat 中单独输出一条组合链 finding**，标注关联的原始 finding 编号，独立定级。**对每条已确认或降级的 finding，主动问一句**：是否可与其它 finding、其它接口或已知攻击面组合成更高危链？避免只做单点验证、忽略组合效应。

### 二、验证结果汇总（在 chat 中直接输出）

全部验证完毕后，**在 chat 窗口** 直接输出汇总，不生成报告文件：

## 后续指令

- **「只验证 Step 1」** → 仅做入口可达性后给出结论
- **「只验证 Step 1～4」** → 不做危害量化与前提现实性，适合快速筛误报
- **「先验证 P0」/「先验证 CRITICAL 和 HIGH」** → 按原始级别优先验证高优先级
- **「只验证 #3,#7」** → 仅验证指定编号的 finding
- **「补充 Finding #XX 的 Step 4」** → 对指定条目标注的待补充信息补全后继续
- **「导出汇总」** → 仅输出汇总表与修复优先级（不逐条展开）
- **「按 BSAF fingerprint 验证」** → 若输入含 fingerprint，验证结论中保留 fingerprint 便于与 BSAF 报告对账
- **「生成 Jira 报告」** → 将确认的 finding 交给「Jira 安全报告」Skill 生成工单（本 Skill 不生成报告文件）
