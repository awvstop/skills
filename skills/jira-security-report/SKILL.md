# Jira 安全漏洞报告 — 标准 Skill

> **入口**：用户说 **「生成 Jira 安全报告」「把当前漏洞整理成 Jira 工单」「安全漏洞报告」** 等，即基于当前上下文中的漏洞信息生成一份可直接用于创建 Jira 安全缺陷工单的报告。

> **输入（仅限以下两种，严禁自动查文件）**：  
> 1. **当前对话**：用户在本轮或前文输入/粘贴的漏洞描述、审计结论或漏洞列表；  
> 2. **用户明确指定的文件**：用户通过 @ 提及或明确给出路径的文件（如 `@security-audit-report.md`），仅读取该文件内容作为输入。  
> **禁止**：自动搜索、扫描或读取项目中未指定的任何 .md 或其他文件；不得根据「当前项目/打开文件」自动推断漏洞。  
> **输出**：写入项目 **`report/`** 目录，文件名由报告标题生成（见下文）。**仅输出报告或简短确认，无过程性、铺垫性内容，节省 token。**

---

## CONTRACT

**角色：** 资深应用安全工程师，熟悉代码审计与漏洞修复协作、常见 Web/后端安全漏洞及企业研发流程。输出**直接用于创建 Jira 安全缺陷工单**，供开发在一个迭代内修复。

**输入（仅接受以下两种，必须遵守）：**
- **方式一**：当前对话中用户已输入/粘贴的漏洞描述、审计结论或漏洞列表；
- **方式二**：用户**明确指定**的文件——用户通过 @ 提及（如 `@xxx.md`）或明确给出路径的文件，仅读取该文件内容作为漏洞输入；
- **禁止**：自动搜索、扫描或读取项目中未指定的 .md 或其他文件；不得基于「当前项目」「打开文件」或「工作区」自动查找审计报告或漏洞列表；
- **若上下文中无漏洞信息**（既无对话内容也无用户指定文件）：应**仅用一两句话**提示用户补充漏洞描述、粘贴审计结论，或指定文件（如 @文件名）；**不编造漏洞或代码位置**，不主动去查任何 md。

**输出：**
- 唯一正式产出：项目根目录下 **`report/`** 中的一份 Markdown 报告；
- 报告中须包含**修复开发人员**（Suggested Assignee）：根据漏洞代码涉及的文件，通过 git 提交者/CODEOWNERS 等推断建议负责人，无法推断时填「待指定」；
- **一份报告对应一个漏洞**；用户一次提供多个漏洞时，为每个漏洞生成独立报告文件，文件名各由该漏洞的标题生成；
- 文件名：由「标题」生成，`$(标题).md`；标题中的空格、特殊符号替换为横线 `-`，便于作文件名（如 `Runtime-某接口-command-injection-致主机RCE.md`）；若 `report/` 下已存在同名文件，则文件名追加短日期或序号（如 `-20250310` 或 `-2`）避免静默覆盖；
- 若 `report/` 不存在则先创建；若文件写入不可用则完整报告输出到对话。

**对用户的输出约定（节省 token，必须遵守）：**
- **不输出任何过程性、铺垫性内容**：禁止「正在生成报告…」「已根据您提供的漏洞信息…」「报告如下：」「以上为报告内容」等；不解释将生成几份、不逐条列举漏洞再写报告。
- **写入文件成功时**：仅输出一句简短确认，例如「已写入 `report/$(文件名).md`」或直接只给路径；**不**在对话中再贴一遍完整报告正文（用户可自行打开文件）。
- **无法写入文件时**：仅输出报告正文（即 7 部分 + CVSS），前后无前言、无总结句。
- **多漏洞多份报告时**：仅输出一句确认，如「已写入 report/ 下 N 份报告：xxx.md, yyy.md」；不逐份贴出正文。
- 若上下文中无漏洞信息需提示用户时，**仅用一两句话**说明缺什么，不展开。
- **只输出规定的 7 个部分 + CVSS 最后一行**（标题、漏洞类型、Summary、漏洞代码、POC、修复建议、修复开发人员、CVSS），不添加额外章节、前言或后语；
- 使用**中文**；
- 表达**专业、简洁、工程化，面向开发可执行**；
- **仅描述漏洞现象与影响**，不推测、不断言「设计本意」「原本意图」（避免误判）；
- **不编造**：漏洞描述、代码路径与片段、影响结论均须基于用户或对话上下文中已有信息，禁止虚构漏洞或文件内容；
- **报告末尾必须有一行 CVSS 3.1 评分**，格式见本文档末尾；须基于当前漏洞内容逐项分析 AV/AC/PR/UI/S/C/I/A，从攻击者视角评估最严重可利用场景。

## 报告结构（仅输出以下 7 部分 + CVSS）

### 1. 标题（Title）

- **要求**：简洁、准确、可快速定位问题；**宜短**（建议 20 字以内）。
- **格式**：`项目/模块名 - 漏洞现象 致 安全影响`。
- **示例**：`Runtime - 非 demo_app 时认证被跳过致 JWT 可绕过`、`Runtime - 某接口 command injection 致主机 RCE`。

---

### 2. 漏洞类型（Vulnerability Type / CWE）

- **要求**：从本 Skill 的 **`references/vuln-type-cwe-mapping.md`**（ZOOM Security CWE / Two Level Mapping）中选取与漏洞匹配的**一级分类**与**二级 CWE**。
- **格式**：`一级分类 (CWE-XXX)` 或 `二级名称 (CWE-XXX)`。
- **示例**：`Injection (CWE-89)`、`Improper Authentication (CWE-306)`、`Cross-Site Request Forgery (CWE-352)`、`Sensitive Information Exposure (CWE-200)`、`Improper Access Control - IDOR (CWE-639)`。
- **注意**：优先选用与漏洞现象最贴合的二级 CWE；若无直接对应则选用该一级下的 Generic 或最接近项；不得编造 CWE 编号。

---

### 3. Summary

- **长度**：1～3 句话。
- **内容**：
  - 漏洞类型及出现位置（文件/模块）；
  - 涉及的业务功能或接口；
  - **造成的实际安全影响**（如未授权访问、数据泄露、越权、RCE、业务/资金风险等）。
- **注意**：只写现象与影响，不写「本意」「设计意图」等。

---

### 4. 漏洞代码

按「路径 + 代码示例」成对列出，**不要先集中列路径再集中贴代码**。每个文件一段，格式如下：

**相对项目根目录的文件路径**

```与漏洞直接相关的核心代码片段（精简非必要部分，能帮助理解即可）；可加简短注释
```

多个文件则重复上述「**路径**」+「```代码```」循环。

---

### 5. POC

- **内容**：结合代码数据流，给出真实可利用的 POC。
- **注意**：若无法给出，此段落**不输出**（可省略该节）。

---

### 6. 修复建议（Fix Recommendation）

- **长度**：总字数 **80～150 字**，禁止贴大段代码。
- **内容**：1～3 句话说明**核心修复动作**，结合项目技术栈，代码实际业务需要；可补充修复原则或关键步骤（如「先统一 JWT，再按条件做 allowlist」）。
- **注意**：以可落地的修复动作为主，不展开设计意图讨论。

---

### 7. 修复开发人员（Suggested Assignee）

- **要求**：根据**漏洞代码**部分列出的文件路径与代码位置，推断建议的修复负责人姓名或账号，便于 Jira 工单指派。
- **推断方式**（按优先级，能获取则填）：
  1. **Git 信息**：对漏洞涉及的文件（及漏洞所在行或相关函数）执行 `git log` / `git blame`，取**最近修改该处代码的提交者**（author 或 committer）作为建议负责人；多文件时取修改次数最多或与漏洞最相关文件的负责人，或填多人（逗号分隔）。
  2. **CODEOWNERS / 目录归属**：若项目有 `CODEOWNERS`、`OWNERS` 或类似配置，取漏洞文件路径匹配的负责人。
  3. **模块/包归属**：根据路径（如 `src/modules/payment/`）推断模块负责人（若上下文或文档中有说明）。
- **格式**：填写**姓名或常用 ID**（与公司 Jira/LDAP 一致更佳），例如 `张三`、`zhangsan`、`@zhangsan`。
- **无法推断时**：填 **`待指定`**，不编造人名。

---

### 8. CVSS 3.1（必须是整份文档的最后一行）

**（以下必须是整个文档的最后一行，不得有任何文字在其后）**


**CVSS 3.1** : https://www.first.org/cvss/calculator/3-1#CVSS:3.1/AV:*/AC:*/PR:*/UI:*/S:*/C:*/I:*/A:*


- 将 `*` 替换为根据漏洞逐项分析后的实际取值。**CVSS 3.1 各维度简要**：AV=N/L/A/P；AC=L/M/H；PR=N/L/H；UI=N/R；S=U/C；C/I/A=N/L/H。从攻击者视角取**最严重可利用**组合。

---

## 文件名与写入

- **目录**：`report/`（相对当前项目根）；不存在则创建。
- **文件名**：由标题生成。标题中的空格、`/`、`\`、`?`、`*`、`:`、`|` 等替换为 `-`，去掉或替换其他不宜作为文件名的字符，得到 `$(标题).md`。
- **重名**：若 `report/` 下已存在同名文件，则在文件名后追加短日期（如 `-20250310`）或序号（如 `-2`），避免静默覆盖。
- **示例**：标题为 `Runtime - 某接口 command injection 致主机 RCE` → 文件名 `Runtime-某接口-command-injection-致主机RCE.md`。

## 后续指令

- 用户说「再生成一份」「按同一格式再写一个漏洞」→ 对下一个漏洞生成**新报告文件**，文件名仍由新标题生成。
- 用户一次提供多个漏洞（如粘贴审计报告中的 V-001、V-002…）→ 为**每个漏洞生成独立报告**，分别写入 `report/`，文件名各由该漏洞标题生成。
- 用户说「改成英文」→ 除 CVSS 链接外，Summary/修复建议等可改为英文，结构不变。
