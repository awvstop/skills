# Jira 安全漏洞报告 — 标准 Skill

**入口**：用户说「生成报告」「生成 Jira 安全报告」「把当前漏洞整理成 Jira 工单」「安全漏洞报告」等，即基于上下文漏洞信息生成可落 Jira 的 Markdown。**输入、禁止项、文件命名与对话输出约定以 CONTRACT 为准。**

---

## CONTRACT

**角色：** 资深应用安全工程师，熟悉代码审计与漏洞修复协作、常见 Web/后端安全漏洞及企业研发流程。输出**直接用于创建 Jira 安全缺陷工单**，供开发在一个迭代内修复。

**输入（仅接受以下两种，必须遵守）：**
- **方式一**：当前对话中用户已输入/粘贴的漏洞描述、审计结论或漏洞列表；
- **方式二**：用户**明确指定**的文件（@ 提及或明确路径，如 `@security-audit-report.md`），仅读取该文件内容作为漏洞输入；
- **禁止**：自动搜索、扫描或读取项目中未指定的 .md 或其他文件；不得基于「当前项目」「打开文件」「工作区」自动推断或查找审计报告；
- **若上下文中无漏洞信息**（既无对话内容也无用户指定文件）：**仅用一两句话**提示用户补充漏洞描述、粘贴审计结论或指定文件；**不编造**漏洞或代码位置，不主动查任何 md。

**输出（文件与内容）：**
- 唯一正式产出：项目根目录 **`report/`** 下 Markdown；目录不存在则创建；**写入失败**时完整报告输出到对话。
- **一份报告 = 一个漏洞**；多漏洞时各写独立文件。文件名由该漏洞**标题**生成 `$(标题).md`：标题中空格及 `/` `\` `?` `*` `:` `|` 等替换为 `-`，去掉或替换其他不宜作文件名的字符（示例：`Runtime - 某接口 command injection 致主机 RCE` → `Runtime-某接口-command-injection-致主机RCE.md`）；**重名**时追加短日期（如 `-20250310`）或序号（如 `-2`）避免静默覆盖。
- 报告须含**修复开发人员（Suggested Assignee）**；推断优先级、格式与「待指定」规则见下节「修复开发人员」。

**对用户的对话输出（节省 token，必须遵守）：**
- **禁止**过程性、铺垫性话术（如「正在生成…」「报告如下：」「以上为报告内容」等）；不解释将写几份、不先逐条列举漏洞再写报告。
- **写入成功**：仅一句短确认（如「已写入 `report/$(文件名).md`」）或只给路径；**不在对话中再贴**完整报告正文。
- **写入失败**：仅输出报告正文（7 部分 + CVSS），前后无前言、无总结句。
- **多份报告**：仅一句列出路径，如「已写入 report/ 下 N 份报告：xxx.md, yyy.md」。
- **报告正文**：仅 **7 部分 + CVSS 最后一行**（结构见下），不增删章节；**中文**；专业、简洁、工程化、面向开发可执行；**仅描述现象与影响**，不断言「设计本意」；**不编造**描述、路径、代码片段与影响结论。
- **CVSS**：须为整份报告的**最后一行**，格式见本文档末尾；逐项分析 AV/AC/PR/UI/S/C/I/A，从攻击者视角取结合实际可利用性，造成的实际危害取最贴合实际的值。

## 报告结构（仅输出以下 7 部分 + CVSS）

### 标题（Title）

- **要求**：简洁、准确、可快速定位问题；**宜短**（建议 20 字以内）。
- **格式**：`[项目] 模块名 - 漏洞现象 致 安全影响`。
- **示例**：`[Synora] Runtime - 非 demo_app 时认证被跳过致 JWT 可绕过`、`[Synora] Runtime - 某接口 command injection 致主机 RCE`。。

---

### 漏洞类型（Vulnerability Type / CWE）

- **要求**：从本 Skill 的 **`references/vuln-type-cwe-mapping.md`**（ZOOM Security Two Level Mapping）中选取与漏洞匹配的**一级分类**与**二级分类**：**一级**为各章标题中的分类名（含标题括号内的 CWE，若有）；**二级**为表中与现象最贴合的一行（「二级 CWE」列：编号 + 说明即二级分类名称）。
- **格式**：`一级分类 / 二级分类 (CWE-XXX)`，**CWE-XXX** 须与所选二级分类在表中的编号一致。
- **示例**：`Injection / SQL Injection (CWE-89)`、`Improper Authentication / Missing Authentication for Critical Function (CWE-306)`、`Cross-Site Request Forgery / Cross-Site Request Forgery (CSRF) (CWE-352)`、`Sensitive Information Exposure / Information Exposure Through an Error Message (CWE-209)`、`Improper Access Control / Insecure Direct Object Reference (IDOR) (CWE-639)`。
- **注意**：优先选用与漏洞现象最贴合的**二级分类**；若无直接对应则选用该一级下的 Generic 或最接近项；不得编造 CWE 编号。

- **硬性校验（必须）**：生成报告文件前，`漏洞类型` 必须且只能有一行，且严格匹配 `一级分类 / 二级分类 (CWE-XXX)`；`CWE-XXX` 必须与所选二级分类编号一致。
- **禁止项**：禁止输出不在映射表中的自造分类；禁止仅写 `CWE-XXX` 或仅写一级分类；禁止使用模糊占位（如 `Misc.`、`其他`）代替已存在映射。

---

### Summary

- **长度**：1～3 句话。
- **内容**：
  - 漏洞类型及出现位置（文件/模块）；
  - 涉及的业务功能或接口；
  - **造成的实际安全影响**（如未授权访问、数据泄露、越权、RCE、业务/资金风险等）。
- **注意**：只写现象与影响，不写「本意」「设计意图」等，注意适时分段便于阅读。

---

### 漏洞代码

按「路径 + 代码示例」成对列出，**不要先集中列路径再集中贴代码**。每个文件一段，格式如下：

**相对项目根目录的文件路径**

```与漏洞直接相关的核心代码片段（精简非必要部分，能帮助理解即可）；漏洞点加简短注释
```

多个文件则重复上述「**路径**」+「```代码```」循环。

---

### POC

- **内容**：结合代码数据流，给出真实可利用的 POC/EXP，参数要结合代码分析，真实不捏造。如果poc分多个步骤，按照步骤123这样列清楚。
- **注意**：若无法给出，此段落**不输出**（可省略该节）。

---

### 修复建议（Fix Recommendation）

- **长度**：总字数 **80～150 字**，禁止贴大段代码。
- **内容**：1～3 句话说明**核心修复动作**，结合项目技术栈，代码实际业务需要；可补充修复原则或关键步骤（如「先统一 JWT，再按条件做 allowlist」）。
- **注意**：结合业务实际需求，以可落地的修复动作为主，不展开设计意图讨论。

---

### 修复开发人员（Suggested Assignee）

- **要求**：根据**漏洞代码**部分列出的文件路径与代码位置，推断建议的修复负责人姓名或账号，便于 Jira 工单指派。
- **推断方式**（按优先级，能获取则填）：
  1. **Git 信息**：对漏洞涉及的文件（及漏洞所在行或相关函数）执行 `git log` / `git blame`，取**最近修改该处代码的提交者**（author 或 committer）作为建议负责人；多文件时取修改次数最多或与漏洞最相关文件的负责人，或填多人（逗号分隔）。
  2. **CODEOWNERS / 目录归属**：若项目有 `CODEOWNERS`、`OWNERS` 或类似配置，取漏洞文件路径匹配的负责人。
  3. **模块/包归属**：根据路径（如 `src/modules/payment/`）推断模块负责人（若上下文或文档中有说明）。
- **格式**：填写**姓名或常用 ID**（与公司 Jira/LDAP 一致更佳），例如 `张三`、`zhangsan`、`@zhangsan`。
- **无法推断时**：填 **`待指定`**，不编造人名。

---

### CVSS 3.1（必须是整份文档的最后一行）

**（以下必须是整个文档的最后一行，不得有任何文字在其后）**


**CVSS 3.1** : https://www.first.org/cvss/calculator/3-1#CVSS:3.1/AV:*/AC:*/PR:*/UI:*/S:*/C:*/I:*/A:*


- 将 `*` 替换为根据漏洞逐项分析后的实际取值。**CVSS 3.1 各维度简要**：AV=N/L/A/P；AC=L/M/H；PR=N/L/H；UI=N/R；S=U/C；C/I/A=N/L/H。从攻击者视角取结合实际可利用性，造成的实际危害取最贴合实际的值。

## 生成前自检（必做）

- `漏洞类型` 是否严格为 `一级分类 / 二级分类 (CWE-XXX)` 单行格式。
- `一级分类` 与 `二级分类` 是否均来自 `references/vuln-type-cwe-mapping.md`。
- `CWE-XXX` 是否与所选二级分类编号完全一致。
- 对话回复是否遵循 CONTRACT：写入成功仅回路径，不回贴正文。
- 若用户指出格式不符合：必须覆盖修正原报告文件，并在对话仅回复修正后的路径。


- 用户说「再生成一份」「按同一格式再写一个漏洞」→ 对下一个漏洞生成**新报告文件**，文件名仍由新标题生成。
- 用户一次提供多个漏洞（如粘贴审计报告中的 V-001、V-002…）→ 为**每个漏洞生成独立报告**，分别写入 `report/`，文件名各由该漏洞标题生成。
- 用户说「改成英文」→ 除 CVSS 链接外，Summary/修复建议等可改为英文，结构不变，语法正确，安全专业。
