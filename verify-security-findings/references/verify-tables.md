## 源码审阅硬门槛（避免「扫描说啥信啥」）

- **扫描/BSAF 输出 = 待证假设**：classification、grep 命中、LLM 摘要均**不能**单独支撑 `✅确认`。
- **确认漏洞前**：Sink 文件片段、用户输入如何到达 Sink（或为何不可达）、关键防护实现**至少一处**须来自**当前仓库中已读原文**。
- **判误报前**：否定理由须指向**已读到的**具体实现（如「已读 `X.java:120-180` 全分支均为 `PreparedStatement`」），而非泛泛「应该安全」。
- **行号漂移**：以当前文件为准；若与报告不一致，在依据中说明。
- **读不到代码**：`⏸️存疑`，列出路径；**禁止**在零源码情况下 `✅确认`。

## Step 1 — 入口可达性按应用类型快速判断

| 应用类型 | 检查要点 |
|----------|----------|
| Web API | 路由注册表 + 中间件链 + 网关配置 |
| 桌面应用 | 监听地址(localhost?) + IPC机制 + CSWSH |
| 移动端 | deeplink/intent暴露面 + WebView桥接 |
| 微服务内部 | 是否仅限 service mesh 内部调用；消息队列/事件总线（Kafka/SQS/RabbitMQ 等）需确认是否存在外部可注入的消费路径（Consumer ACL、Topic 权限、公开 Webhook 触发），无则视为内部可达 |
| CLI工具 | 输入来源是否外部可控（参数/stdin/环境变量） |
| Serverless | 触发器类型(HTTP/事件) + API Gateway配置 |

**补充判定**：仅通过 `本地脚本`、`CI Job`、`评测Runner`、`人工运维命令` 触发的路径，不满足外部入口条件，默认不得判 `✅确认`。

**特性开关 / 死代码**：结论涉及「生产是否执行该路径」时，结合 feature flag、编译裁剪、路由注册条件；证据不足标 ⏸️。

### Step 1 补充 — 微服务内部识别（强制，含 S2S 识别信号）

当被审计组件为微服务架构中的一个服务时，**不得仅因其具有 HTTP 端点和 JWT 认证就默认外部可达**。须依据以下信号综合判定：

**S2S（内部服务）识别信号**（满足 ≥2 条强信号 → 视为"仅内部可达"，须有代码/配置证据）：

| 信号类型 | 识别方式 | 强度 |
|----------|----------|------|
| 认证中间件命名/类型 | 类名含 `Service`、`Internal`、`S2S`、`Machine`（如 `ServiceTokenMiddleware`、`InternalAuthMiddleware`），区别于 `UserAuth`、`SessionAuth`、`OAuth2` | 强 |
| Token 类型字段 | JWT 中 `typ` 字段为 `service`、`srv`、`machine`、`zm_svr` 等非用户类型；或校验逻辑区分用户 token 与服务 token 并仅接受后者 | 强 |
| 调用方代码追踪 | 在仓库内搜索该服务 URL/base_url/endpoint 的引用，仅出现在其他后端服务的 client/SDK 中，无前端/BFF/Gateway 调用 | 强 |
| 无 Ingress/Gateway 路由 | 部署配置（Helm values、Ingress yaml、API Gateway 路由表、nginx conf）中无该服务的对外暴露条目 | 强 |
| 服务间 client 封装 | 存在专用 `XxxClient`/`XxxServiceClient` 类供其他服务调用，无对外 SDK 或公开文档 | 中 |
| 网络策略隔离 | NetworkPolicy / Security Group 限制仅允许特定服务 IP/namespace 访问 | 强 |
| 端口/协议非标准 | 监听非 80/443 端口，或使用 gRPC/Thrift 等非 REST 协议 | 弱 |

**反向信号**（存在以下任一 → 不可简单判"内部可达"，须继续 Step 2+）：

| 反向信号 | 说明 |
|----------|------|
| API Gateway 有路由映射 | Ingress/Kong/Nginx 将外部请求路由到该服务 |
| 前端/BFF 代码中有调用 | 用户请求可通过 BFF 透传到达该服务 |
| 文档或 OpenAPI 声明公开 | README/Swagger 标注为外部可调用 |
| 认证同时接受用户 token | 同一中间件同时处理 `zm_user` 和 `zm_svr` 类型 token |

**判定流程**：
1. 命中 ≥2 条强信号且无反向信号 → 内部可达，Step 1 退出为 ⚠️降级/💡KNOWN RISK
2. 命中强信号但存在反向信号 → ⏸️存疑，列明冲突证据
3. 无强信号 → 按默认"外部可达"继续 Step 2

**对 IDOR/越权类 finding 的特殊要求**（S2S 场景下）：即使 Step 1 判为"内部可达"导致降级，若 finding 声称跨用户/跨租户，**须额外验证上游调用方是否对关键标识符做了归属绑定**（见 Step 5 IDOR 补充）。上游已绑定 → 💡KNOWN RISK（纵深防御建议）；上游未绑定 → 维持原级或仅降一级。

## Step 2 — 信任边界层级定义

| 层级 | 描述 | 典型漏洞 |
|------|------|----------|
| L1 | 匿名用户 → 认证用户 | 认证绕过 |
| L2 | 普通用户A → 普通用户B | 水平越权 |
| L3 | 普通用户 → 管理员 | 垂直越权 |
| L4 | 客户端 → 服务端 | Client-Server 边界 |
| L5 | 外部服务 → 内部服务 | 网络边界 |
| L6 | 应用层 → 基础设施层 | 沙箱/容器逃逸 |
| L7 | 租户A → 租户B | 多租户隔离 |

## Step 3 — Source 可控性分级

| 级别 | 含义 | 典型证据 | 可利用性 |
|------|------|----------|----------|
| 完全可控 | 攻击者可直接注入请求体/参数/Header | req.body/query/params 直传 Sink | 高 |
| 部分可控 | 长度/字符集/格式受限但仍可构造 payload | 过滤后仍可注入关键字符 | 中 |
| 间接可控 | 经存储/转发/二次渲染后到达 Sink | 存储型 XSS、二次 SQLi、RAG 投毒 | 须追踪完整链 |
| 强约束可控 | 白名单/枚举/强类型后无法注入 | 仅枚举值、严格 schema | 不可控→FP |
| 不可控 | 来源为常量/配置/内部服务且不可伪造 | 非用户输入 | FP |

**间接路径（强制）**：判 **不可控** 或 **❌FP** 前，须排除 **数据库/缓存回显、消息队列、异步任务、Webhook 回调、模板二次渲染、RAG/Agent 上下文**等使外部或低信任数据日后到达 Sink 的路径。仅看同步 HTTP 入参**不够**；未追链 → **⏸️**，不判 FP。

## Step 4 — 天然断点及其失效条件（必须检查）

| 天然断点 | 失效条件 |
|----------|----------|
| 类型转换（string→int/bool/enum） | 转换前已拼接进危险上下文；宽松类型语言的隐式转换 |
| ORM参数化 / Prepared Statement | `.raw()` / `.extra()` / 字符串拼接进 QuerySet；动态表名列名 |
| 模板引擎自动转义（Jinja2/React等） | `\|safe` 过滤器 / `Markup()` / `{% autoescape false %}` / `dangerouslySetInnerHTML` / `href="javascript:"` |
| 序列化/反序列化重构 | 自定义反序列化器 / 宽松的 JSON Schema / 缺少 additionalProperties: false |
| 框架级CSRF防护 | 特定 Content-Type 豁免 / `@csrf_exempt` / SameSite=None + 第三方上下文 |
| Content-Type强制校验 | 某些框架对空 Content-Type 的 fallback 行为；`text/plain` CORS 简单请求 |
| 路径规范化（realpath） | TOCTOU竞争 / 先拼接后规范化（顺序错误）/ 符号链接未处理 |
| 编译期类型安全（Rust/Go） | unsafe 块 / cgo 调用 / 反射 / 泛型 interface{} |
| 长度截断 / 字符集限制 | 截断后仍保留危险语义（SQL截断攻击）；多字节字符集绕过 |

## Step 5 — 按漏洞类别的验证要点

| 漏洞类别 | 关键验证点 |
|----------|-----------|
| **SQLi** | ORM vs raw query + 参数化是否贯穿所有路径 + 动态表名/列名/ORDER BY处理 |
| **XSS** | 输出上下文(HTML/JS/URL/CSS) + 模板引擎转义覆盖范围 + DOM-based场景 + CSP策略 |
| **SSRF** | 协议限制(仅http/https?) + 内网IP黑名单(含IPv6/短格式/DNS rebinding) + 重定向跟随 |
| **路径遍历** | 规范化顺序(先规范化后拼接?) + 根路径jail + 符号链接跟随 + 空字节截断(老版本) |
| **反序列化** | 反序列化类型(原生/JSON) + gadget chain可用性 + 类型白名单 + 签名校验 |
| **IDOR** | 资源标识符可预测性 + 对象级权限校验(非仅认证) + 批量枚举可行性 + **S2S 场景须验证上游调用方归属绑定**（见下方 IDOR 补充） |
| **命令注入** | 调用方式(shell=True?) + 参数数组 vs 字符串拼接 + 环境变量注入 |

### IDOR 补充 — S2S / 微服务架构下的归属校验验证（强制）

当 IDOR finding 所在服务已在 Step 1 被判定为"仅内部可达"（S2S）时，**不可仅凭"下游服务缺少 ownership check"就确认 HIGH/CRITICAL**。须执行以下补充验证：

**必须回答的问题**：终端用户能否通过上游服务（API Gateway / BFF / 调用方服务）将**攻击者控制的资源标识符**传递到下游 Sink？

**验证步骤**：

1. **识别唯一调用方**：在仓库内搜索下游服务的 base_url / endpoint / client 类引用，确定所有上游调用方。
2. **追踪关键标识符来源**：在上游调用方代码中，确认传递给下游的资源标识符（如 `sandbox_id`、`resource_id`）是**从用户请求参数直传**还是**从服务端会话/存储中取值**。
3. **判定用户可控性**：
   - 用户请求参数直传 → 用户可控，IDOR 仍成立（维持原级）
   - 从 session/DB/内部状态取值，用户无注入点 → 用户不可控，降级为纵深防御建议

**判定规则**：

| 上游绑定情况 | 用户可控性 | 判定 |
|---|---|---|
| 上游从 session/DB 取值，用户无法指定任意值 | 不可控 | 💡KNOWN RISK（纵深防御：下游宜补 ownership check，但非直接可利用） |
| 上游直接透传用户请求参数到下游 | 完全可控 | 维持原级（IDOR 可利用，仅下游缺校验不影响定级） |
| 上游部分接口透传、部分接口绑定 | 部分可控 | ⚠️降级并注明可利用路径 |
| 无法确认上游行为（跨仓/代码不可读） | 不确定 | ⏸️存疑，列出需读取的上游代码路径 |

**常见误判模式**：
- 看到"服务 X 的接口缺少 user_id 校验"就判 IDOR HIGH，但实际该服务**仅被另一个已做绑定的服务调用**，终端用户根本无法指定任意资源 ID
- 被审计服务的 `user_id` 参数带下划线前缀（如 `_user_id`）或仅用于日志，易误判为"接收但未校验"（实际可能是设计上信任上游已校验）

| 漏洞类别 | 关键验证点 |
|----------|-----------|
| **XXE** | XML解析器配置(外部实体/DTD是否禁用) + 文件协议 + OOB通道 |
| **认证绕过** | 时序攻击 + 密码重置流程 + JWT算法混淆(none/HS256-RS256) + 会话固定 |
| **业务逻辑** | 支付/状态机/权限是否仅前端或可被绕过；关键操作是否服务端二次校验 |
| **竞态/TOCTOU** | 检查→执行之间是否有竞态；并发下权限/余额/库存是否原子校验 |
| **不安全默认** | 默认密钥/调试开关/默认账号、生产是否仍用开发配置 |
| **AI/LLM/Prompt 注入** | 用户输入是否直入 system/user prompt；LLM 输出是否进入 SQL/命令/文件；Tool 是否有 scope/审批 |
| **Prototype Pollution** | 深合并/object.assign 是否接受用户 key；是否影响授权/模板/查询构造 |
| **Mass Assignment** | 是否白名单绑定；req.body 是否直传 Entity；whitelist/fields 是否覆盖敏感字段 |
| **GraphQL** | depth/cost/alias/batch 限制；字段级授权；嵌套 resolver 是否逐层校验 context.user |
| **供应链/依赖 CVE** | 组件是否打进**运行时产物**、是否**实际调用**脆弱 API、版本锁与 overrides；无调用路径可能 INFO/降级而非等同应用 RCE |
| **跨服务/多仓** | 入口与 Sink 分属不同服务时，分段验证并在依据中写明**每段证据**；不跨服假设权限传递 |

## Step 6 — 利用前提现实性

### Part A — 快速降级触发（任一命中→先降级再评估；须有代码/配置证据，禁止主观推断）

| 触发条件 | 默认调整 | 例外（不降级） |
|----------|----------|----------------|
| 仅管理员/高权限账号可触达 | 降级或 FP | 漏洞本身即「管理员账号接管」或「权限提升」 |
| 仅内网/特定网络位置可达 | 降级并注「需内网」 | 内网可达 Sink 仍有横向移动/数据泄露高危价值 |
| 仅调试/开发配置下触发 | 降级并注「生产不触发」 | 已确认生产有节点开启该配置 |
| 仅 CI / 本地评测 / 人工运维可触发 | 降级或 KNOWN RISK | — |
| 利用依赖竞态/精确时序，且无并发放大手段可稳定触发 | 降级并注「利用可靠性低」 | 可批量并发或工具辅助稳定触发的竞态 |
| 需受害者执行操作 + 目标为普通权限页面 | -1 级 | 目标操作为高权限/批量/不可逆（管理员审批、批量转账等） |
| 需多步组合才可利用 | 单条降级，标组合链并独立定级 | — |
| 默认配置下即成立 | 维持原级 | — |

### Part B — 可利用性评估维度（填后按决策规则得出调整）

| 维度 | 选项 |
|------|------|
| 触发方式 | 0-click / 需受害者点击链接 / 需受害者登录后访问特定页面 / 需管理员操作 |
| 攻击者起点 | 匿名 / 普通注册用户 / 特定角色 / 内部人员 |
| 利用可靠性 | 稳定复现 / 需多次尝试（竞态） / 精确时序依赖 |
| 受害者前提 | 无 / 需有特定账号状态或数据 / 需访问特定路径 |
| 环境依赖 | 默认配置即成立 / 非默认配置 / 特定版本或功能开关 |

**决策规则（从上往下匹配第一条成立的规则）**：

1. 触发=0-click + 起点=匿名 + 可靠性=稳定 + 环境=默认 → **维持原级**
2. 触发=0-click + 起点=认证用户 + 可靠性=稳定 → **维持或 -1 级**（视影响范围）
3. 任意触发 + 可靠性=竞态/时序依赖 + 目标≠高权限/不可逆操作 → **-1 级并注「可靠性低」**；若目标为高权限/不可逆操作则维持原级（竞态不抵消高危影响）
4. 需受害者操作 + 目标=高权限/不可逆操作 → **维持原级**
5. 需受害者操作 + 目标=普通权限页面 → **-1 级**
6. 环境依赖=非默认/特定版本 → **-1 级并注「条件性成立」**
7. 多项同时触发（如需受害者操作 + 竞态 + 非默认配置）→ **-2 级或降至 LOW / INFO**

> `-1 级`：CRITICAL→HIGH、HIGH→MEDIUM、MEDIUM→LOW。降级后在【依据】中写明触发规则编号与条件；降至 INFO 以下则考虑 KNOWN RISK 或 FP。

## AI 扫描器常见误报模式

| 误报模式 | 快速判定方法 |
|----------|-------------|
| 忽略认证/授权边界 | 检查路由中间件是否强制认证 |
| 忽略框架自动防护 | 确认ORM参数化、模板自动转义等是否生效，并检查失效条件 |
| 内部函数当外部入口 | 追踪调用链确认有无外部可达路径 |
| 需先有高权限才能利用 | 评估是否为正常功能而非漏洞 |
| 默认配置下不触发 | 确认实际部署配置与报告假设是否一致 |
| 同一模式重复报告 | 确认根因后按防护栏规则批量处理 |
| 将设计行为标记为漏洞 | 对照产品需求确认是否为预期功能 |
| 忽略间接数据流 | 注意存储型攻击路径，AI扫描器可能仅追踪同步调用链 |
| **S2S 服务当用户直接可达** | 识别 ServiceTokenMiddleware / S2S Auth 等信号；确认是否有 Ingress 路由暴露；查看服务消费者是否仅为内部服务 |
| **下游缺校验即判 IDOR** | 微服务中下游信任上游已校验是常见模式；须追踪上游是否已绑定资源标识符与用户关系，而非仅看下游缺 ownership check |

## 防漏报自检（确认真实漏洞不被误判）

- **入口**：是否已实际读取路由/中间件/网关配置，而非假设「内部」？不同环境（dev/staging/prod）是否一致？
- **信任边界**：「仅影响自身」是否严格限定（同一 userId/tenantId/资源所有者）？是否存在枚举或越权访问其他主体？
- **阻断点**：声称的过滤/校验是否覆盖所有分支（错误处理、默认分支、异步回调）？补偿控制（WAF 等）是否默认启用、是否可绕过？
- **组合与二次影响**：该 finding 是否可与其它漏洞或正常功能组合（如信息泄露+SSRF、IDOR+批量接口）？是否需在汇总时做组合链审视？
