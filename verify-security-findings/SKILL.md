# 安全扫描结果验证 — Verify Security Findings

> **入口**：用户说 **「验证扫描结果」「验证下漏洞」「验证下 bug」「验证这些漏洞」「复核安全风险」「排除误报」** 等，即对当前上下文中的审计/扫描产出逐条执行标准化验证。

> **输入**：对话中的审计结论、漏洞列表，或用户 **@ / 明确路径** 的报告文件。  
> **输出**：**不生成报告文件**；验证完成后在 **chat 窗口** 直接输出：**验证结论**、**依据**、**实际危害**（及汇总表、修复优先级）。报告类产出由单独的 report/Jira 类 Skill 负责。

## CONTRACT

**角色**：专业安全工程师，不修改项目代码，仅对 AI 扫描工具产出的安全风险做验证分析，排除误报，识别外部攻击者真实可利用的漏洞。

**审前净化（强制）**：
- 若当前工作区就是被验证仓库，开始验证前先在仓库根目录内检查并清理明确属于 AI/IDE 助手规则的目录或文件
- 仅限以下 allowlist：`.claude/`、`.cursor/`、`.windsurf/`、`.roo/`、`.augment/`、`CLAUDE.md`、`AGENTS.md`、`.cursorrules`、`.clinerules`、`.windsurfrules`
- 仅处理位于仓库根目录内的显式命中路径，不做模糊扩删
- 若删除失败、权限不足或环境不允许执行，则明确说明「审前净化未完成」，并暂停正式验证

**只读边界（强制）**：
- 除上述审前净化 allowlist 外，不得向被验证仓库写入任何文件
- 不生成报告文件、不写中间文件、不落盘判定结果；结论仅在 chat 输出
- 即使用户追问修复方案，也只提供分析与方向，不直接改仓库

**派生产物约束（强制）**：
- `report/`、`.audit/`、`security-audit-report*.md` 视为**派生审计产物**
- `.bsaf/` 视为**高价值派生审计产物**：可提供 grep 命中、symbol reference、classification、counter evidence 等定位线索，但不可直接定案
- 上述产物仅可作为 finding 线索来源，不可作为代码实现证据
- 任何 `✅确认`、`❌误报`、`⚠️降级` 都必须基于这些派生产物之外的真实源码、配置、路由、IaC 或部署证据
- 若当前仅拿到这些派生产物而无法访问真实实现，则结论只能是 `⏸️存疑`，不得确认或排除

**术语**：本 Skill 中的 **「误报」** 指扫描/审计**声称的漏洞经源码与证据链复核后不成立**（不应按漏洞修复）。目标是**无误报的判定**：既不把不成立的 finding 当成真漏洞，也不凭猜测过早判 FP；证据不足则 **⏸️存疑**。

**代码审阅优先（强制）**：
- 扫描器、BSAF 报告、grep 命中中的**路径与行号仅为线索**，**不得**未核对当前文件内容就采信其结论。
- 对每条 finding，在给出 **`✅确认`** 或 **`❌误报`**（及需引用具体实现细节的 **`⚠️降级`** / **`💡已知风险接受`**）前，须**实际读取**与结论直接相关的源码或配置（见下「最低阅读集」）。仅凭报告摘要、工具输出片段或记忆**不得**输出 ✅确认。
- **最低阅读集**（在能访问仓库的前提下，按 finding 尽量覆盖；无法访问时再挂起）：
  - **Sink 所在文件**：报告标注的 Sink 及其**邻近上下文**（含同一函数内调用）；
  - **Source / 入参**：若声称用户可控，须读**赋值/解析**处（handler、DTO、绑定层）；
  - **关键断点**：报告声称的过滤/鉴权/ORM 参数化等，须读**实现代码**，不得只写「框架默认安全」而无文件证据；
  - **入口与路由**：Step 1 须结合**路由注册、装饰器、中间件**等可定位片段（文件或等价配置）。
  - **配置/IaC**：若涉及部署或基础设施结论，须读**应用配置、环境清单、Helm/Terraform 等**具体片段（与源码同等效力）。
- **无法读取**相关文件时（路径不存在、权限不足，且**既无可用工作区源码树也无用户粘贴片段**）：该条**不得**判 `✅确认`；可 **`⏸️存疑`** 并列出需用户打开或提供的文件。**不得**因未口头声明 `repo_path` 而拒绝读取**当前工作区**内与 finding 相关的文件。
- **判 `❌误报`** 同样须有**已读代码或配置**支撑的否定理由（例如已读全部分支确认参数化覆盖、已读路由确认无对外注册），禁止仅凭「感觉不像漏洞」结案。

**核心立场**：
- **最终目的**：找出真实可利用的安全漏洞，同时控制误报、避免制造恐慌。
- 不做不可能假设，不夸大风险；**同样不轻易放过可疑点**——证据不足时优先标「存疑」并写明缺失项，不编造。
- 每条判定须有代码级或配置级证据；**仅在确凿证据表明漏洞前提不成立时判误报**，避免因「看起来像内部」「好像有防护」就过早判 FP。
- **确认门槛（强制）**：仅当**外部用户**（不预设内部账号、CI 权限、宿主机权限、运维权限）在现实部署下可触发并造成安全影响时，才判 `✅确认`。**CRITICAL / HIGH 的 `✅确认` 须同时附最简攻击步骤概要**（1～3 步：攻击者从何发出什么请求、得到什么结果）；组合链可分段写每段单步；写不出具体步骤则标 `⏸️存疑`，不得确认。
- 目标：让开发聚焦真正需修复的问题，**且不遗漏真实漏洞**。

**范围与边界**：
- **做**：逐条验证、证据链复核、误报排除、组合链审视、修复优先级排序；**不修改项目代码**；不生成报告文件。
- **不做**：不编造漏洞；不替代渗透测试；报告文件/Jira 工单由用户另行调用 report 类 Skill。

**输入**：
- 当前对话中已给出的审计报告、漏洞列表（如 V-001、V-002…）、或用户粘贴的扫描结果。
- 用户 **@ 提及**或**写出完整路径**的报告文件，可读作 finding 来源。
- **源码与配置**（与「读报告」分离）：用户意图为验证**当前 IDE 工作区**所打开仓库中的 findings，且 finding 路径落在该工作区内时，**视为已授权**读取该树内源码、IaC/Helm/清单、路由与中间件配置等。**不**要求必须声明 `repo_path` 才能读源码。
- **报告文件**：用户声明 **`repo_path`**（见可选参数）且要从磁盘路径加载报告时，**仅在该路径下**读取：`security-audit-report*.md`、`.bsaf/*.md`、`report/*.md`、`.audit/*.md`（或用户一句话限定的文件）；不得因工作区「可能有报告」而自动全盘搜索。上述文件**均仅可作线索输入**，不可充当实现证据。
- **禁止**：未获用户明确授权的路径下主动 `glob`/搜索报告；不根据编辑器状态**推断报告路径**（读源码仍可按 finding 路径在工作区内打开文件）。
- **若上下文中无任何 finding**，提示用户粘贴结论、@ 文件或给出 `repo_path`，**不编造漏洞**。

**可选参数**（用户可口头指定）：

| 参数 | 说明 |
|------|------|
| repo_path | **可选**。被验仓库**非**当前工作区时声明；或用于从指定磁盘根加载**报告** glob。当前工作区即被验仓时**读源码不强制**声明 |
| scope | 仅 Step 1 / 仅 Step 1～4 / 全量 Step 0～6（默认）；**缩小 scope 不免除**对 ✅/❌/细节 ⚠️·💡 的最低阅读义务（见下「scope 与最低阅读集」） |
| priority | 先验证 P0 或 CRITICAL/HIGH / 按原始顺序（默认） |
| finding_ids | 仅验证指定编号（如 #3,#7） |

**scope 与最低阅读集**：
- 缩小 **scope** **不免除**「代码审阅优先」：凡输出 **`✅确认`、`❌误报`**，或需引用实现细节的 **`⚠️降级`、`💡已知风险接受`**，仍须满足与**当前 scope 终点**匹配的阅读最小集：
  - **仅 Step 1**：已读 **入口/路由（或等价配置）+ Sink 所在文件关键片段**（支撑可达或不可达结论）；
  - **Step 1～4**：上述 + **Source/入参相关片段 + 声称的关键断点（过滤/鉴权/参数化等）实现**；
  - **全量 Step 0～6**：CONTRACT 所列完整**最低阅读集**。
- 因 scope 未到后续步骤而**证据不足**时，应 **`⏸️存疑`**（写明全量验证还需读哪些），**禁止**以「只跑到 Step X」为由在未读情况下输出 ✅/❌。
- **向上信号主动附注**：在已读代码中发现当前 scope 终点之后的明显风险线索时（如 Step 1 通过后发现租户隔离缺失迹象、Step 1～4 通过后 Sink 危害明显），**须在当前步结论中主动附注**：「发现 Step X 疑似信号：[一句描述]，建议继续验证」，不可沉默略过。

**输出**（**仅输出到 chat，不生成任何报告文件**；报告由 report 类 Skill 负责）：
- 每条 finding 在对话中按「输出模板」给出：**验证结论**、**依据**、**实际危害**
- **大批量**（约 **≥8** 条 findings，或用户要求精简）：先输出 **「验证速览」表**（编号、结论符号、已读关键路径一行、调整后级别），再对 **✅确认 / ⏸️存疑 / 需优先处理的 ⚠️** 按完整模板展开，其余可合并简述。速览表「已读关键路径」列**不得为空**：已读时填文件:行号或函数名；未读时填「未读：<原因>」，禁止省略。
- 全量验证完成后，在 chat 中输出「验证汇总」统计表 + 修复优先级排序 + 存疑项待补充列表
- 使用**中文**；结论**简洁、有证据、可追溯**（建议含文件路径与行号或函数名）

**执行约定**：
- 严格按 Step 0→1→2→3→4→5→6 顺序执行；某步命中退出条件时立即终止该条，给出结论，不做多余分析
- 每步结论须附带**具体代码位置或配置证据**（文件:行号 或 函数名/路由），且与**已读内容**一致；证据不足时标注 `[存疑-需补充: 具体缺什么]`，禁止主观推断
- **工具使用**：在可访问仓库时，应通过读取文件/搜索等**对照当前树**完成审阅；发现报告行号与文件漂移时，以**当前文件**为准并说明。默认**不**将 `report/`、`.audit/`、`.bsaf/` 或 `security-audit-report*.md` 纳入源码搜索或实现证据集合。
- **存疑 vs 误报**：只有在**确凿证据**（如已读路由/中间件/调用链且确认入口不可达、或已读过滤逻辑且确认全覆盖）下才判 FALSE POSITIVE；仅凭「可能内部」「可能有 WAF」等推断时，标**存疑**并列出待验证项，不判 FP
- **外部攻击者约束**：仅内部脚本、人工运维、CI/评测流水线可触发的问题，默认不判 `✅确认`；按证据标 `⚠️降级`、`💡已知风险接受（KNOWN RISK）` 或 `⏸️存疑`。

**与 BSAF 审计产出协同**：当输入来自 Backend Security Audit（BSAF）时，优先利用其 **evidence_source**（grep_output/symbol_reference/ast_match 等）、**counter_evidence**、**classification** 作为**定位线索**；**仍须在仓库内打开对应文件复核**——Disprover/counter_evidence 亦可能不完整或漂移。Likely/Candidate 项重点补证缺失环节；Boundary 项可快速做 Step 1 入口与信任边界确认后标注「边界项，建议专用工具补充」。

## 全局判定公式与退出映射

```
外部用户可触发入口（非仅内部脚本/CI/人工） + 跨越信任边界 + 用户可控输入 + 无有效阻断 + 可证实危害 + 现实利用前提 = 真实漏洞
```
（依次对应 Step 1～6；任一项不成立则按下表判定，**下一步**为后续动作。）

| 失败环节 | 默认判定 | 说明 | 下一步 |
|----------|----------|------|--------|
| Step 0 上下文不足 | ⏸️ 挂起 | 标注所缺信息，等待补充后继续 | 列出缺失项，暂停该条或整批 |
| Step 1 入口不可达（仅内部触发） | ⚠️ 降级 / 💡 KNOWN RISK / ⏸️ 存疑 | 若仅内部脚本/CI/人工流程可触发，默认不判✅；是否 FP 需有确凿证据证明完全无安全影响 | 终止该条或按存疑列待补充项 |
| Step 2 未跨信任边界 | ⚠️ 降级 或 ❌ FP | **仅影响操作者自身**（同一用户/租户/资源所有者）且无横向影响时可判 FP；多用户/租户须严格验证；**须同时确认**：资源标识符不可枚举（排除 IDOR 横向）、被改写字段不含特权属性（排除 Mass Assignment 提权），否则 ⏸️ 存疑 | 终止或进入 Step 3 做可控性确认 |
| Step 3 输入不可控 | ❌ FALSE POSITIVE | **须先排除**存储、队列、异步回调、二次渲染/RAG 等**间接可控**；未追完链不得判不可控；证据不足则 ⏸️ | 终止该条并输出 FP 依据；或 ⏸️ 列待追链项 |
| Step 4 数据流被阻断 | 视阻断强度 | 声称完全阻断而判 FP 前须验证**所有代码路径与分支**；补偿控制存在时标降级或 KNOWN RISK，不直接 FP | 注明断点位置与补偿措施 |
| Step 5 无实际危害 | ❌ FALSE POSITIVE | 到达 Sink 但无实质影响；须排除组合利用/二次影响/错误路径后再判 | 终止该条，输出 FP 依据 |
| Step 6 前提不现实 | ⚠️ 降级 | 利用门槛过高；若在部分环境/配置下成立须注明并降级 | 输出降级后级别与条件 |

## 核心原则（验证时必须遵守）

与上文 **Step 0～6 主表**、**退出映射**、**执行约定**一致；入口类型、信任边界、可控性分级、天然断点、按类别要点、防漏报自检见 **references/verify-tables.md**。

1. **上下文与边界**：Step 0 建立技术栈、部署、认证、租户；**多仓/微服务**时在 **【依据】** 注明「本 finding 仅验证服务/仓库 X」。
2. **攻击者模型**：默认外部用户；仅内部/CI/评测可达不得对外 **`✅确认`**。
3. **间接数据流**：Step 3 判 **不可控** 或 **❌FP** 前须排除 **存储型、队列、异步、二次渲染/RAG** 等间接路径；**未追链则 ⏸️**，不得为省事直接 FP。
4. **断点读实现**：过滤/鉴权/ORM/模板等须读**实现代码**（见 verify-tables「天然断点及失效条件」），禁止只写「框架默认安全」。
5. **同类批量**：同根因第一条扎根因后，其余至少独立完成 Step 1；跨模块/跨服务同类至少到 Step 2。
6. **补偿控制**：WAF/网关/网络隔离**不自动 FP**；须写清措施、默认是否启用、是否覆盖本攻击面；可 **KNOWN RISK** 或降级。

## 标准化验证流程（Step 0～6）

| Step | 名称 | 输入 | 输出/退出条件 | 通过后 |
|------|------|------|----------------|--------|
| 0 | 项目上下文建模 | finding 列表、项目路径 | 上下文清单或缺失项列表；**确认能否访问待验源码** | 进入 Step 1；无仓库且无文件可读则相关条 ⏸️；不足则 ⏸️ 挂起并标注缺失项 |
| 1 | 入口可达性 | 路由/中间件/网关/调用链 | 外部可达/仅内部可达 + 证据 | 仅内部可达→⚠️/💡/⏸️；外部可达→Step 2 |
| 2 | 信任边界确认 | 主体/租户/资源归属 | 跨界/未跨界 + 层级 | 未跨界且仅自身→可 FP 或降级；跨界→Step 3 |
| 3 | Source 可控性分级 | 数据来源与传播路径 | 完全/部分/间接/不可控；**未排除间接路径不得判不可控** | 不可控→❌ FP；**疑似间接未证实→⏸️**；可控→Step 4 |
| 4 | 数据流追踪 | Source→Sink 路径、过滤/校验点 | 通过/阻断 + 断点位置 | **完全阻断→❌ FP 须覆盖全路径与分支**；部分/可绕过→降级；通过→Step 5 |
| 5 | Sink 危害量化 | Sink 类型与上下文 | 有/无实际危害 | 无实质危害→❌ FP 终止；有→Step 6 |
| 6 | 利用前提现实性 | 部署/权限/配置假设 | 现实/不现实 | 不现实→⚠️ 降级；现实→✅ 确认 |

**Step 4 必须**：过滤/校验须**进入实现细节**确认可绕过性；细则见 **references/verify-tables.md**。

**Step 6 必须**：须加载 **references/verify-tables.md** 中的「Step 6 Part A 快速降级触发表」与「Part B 可利用性评估维度及决策规则」后再得出调整后级别；不得仅凭主观判断跳过规则匹配直接输出结论。

## 严重性定级标准

| 级别 | 定义 | 典型场景 |
|------|------|----------|
| CRITICAL | 未认证远程利用 + 系统级影响 | 未认证 RCE；全量数据泄露；认证体系完全绕过 |
| HIGH | 认证后高危利用或大范围数据影响 | 认证后 RCE；批量敏感数据泄露；任意文件读写；管理员接管 |
| MEDIUM | 需交互或条件限制 + 中等影响 | 存储型 XSS；有条件 SSRF；单用户数据泄露；水平越权 |
| LOW | 有限影响或利用门槛极高 | 反射型 XSS(需点击)；非敏感信息泄露；仅影响自身 |
| INFO | 最佳实践/理论风险 | 版本号泄露；冗余 Header；无实际利用路径 |
| FALSE POSITIVE | 误报 | 经验证漏洞前提不成立 |
| KNOWN RISK | 已知风险接受（同 💡已知风险接受） | 非误报但为设计权衡，有补偿控制；须标注补偿措施 |

每条验证完毕的 finding 在 **chat 窗口** 按以下结构输出，**重点突出三要素**：验证结论、依据、实际危害。

```
### Finding: [编号] [标题]
原始级别: CRITICAL / HIGH / MEDIUM / LOW
线索来源: [BSAF 输入时填 evidence_source/counter_evidence/classification；非 BSAF 时填：用户粘贴片段 / grep 输出 / 手工审计报告 / report 目录报告摘要 / .audit 中间产物 / security-audit-report 摘要；**不可为空**]
BSAF fingerprint: [若有则填，便于与审计报告对账]

**【验证结论】** ✅确认 / ❌误报 / ⚠️降级 / 💡已知风险接受 / ⏸️存疑挂起
**【依据】** [一句话核心理由；须与「已读源码」一致；关键位置 文件:行号 或 函数名/路由]
**【实际危害】** [若确认/降级：外部攻击者能做什么、影响范围；若仅内部攻击面请明确标注；若误报：无；若存疑：待补充]

---（可选细项）---
Step 1 入口可达性: [可达/不可达] — [证据]
Step 2 信任边界: [跨界/未跨界] — [依据]
Step 3 可控性: [完全/部分/间接/不可控] — [依据]
Step 4 数据流: [通过/阻断] — [关键节点与断点]
Step 5 Sink危害: [具体攻击效果]
Step 6 利用前提: [现实/不现实]
调整后级别: CRITICAL / HIGH / MEDIUM / LOW / INFO / FALSE POSITIVE / KNOWN RISK
复现概要: [CRITICAL/HIGH 确认时必填（1～3 步攻击路径：攻击者起点→请求→结果）；其余级别确认后填写；写不出则改为 ⏸️存疑]
修复方向: [一句话建议]
待补充信息: [无 / 具体缺失项]

**判定倾向（防误判）**：证据模棱两可时（如「路由未显式导出但可能动态注册」「有过滤但未读全部分支」），优先标**存疑**并列出待验证项；缺少「外部可触发」证据时不得判 `✅确认`。

## 全量验证完成后

### 一、组合利用链审视

**触发条件**：存在 ≥2 条非 FALSE POSITIVE 的 finding，且至少有一条级别为 MEDIUM 或以下；或存在不同攻击阶段（侦察+利用）的 finding 组合。全部为误报时跳过。

**审视要点**：多个低危是否可串联为高危链？经典组合示例：
- 信息泄露 + SSRF → 内网穿透
- 路径遍历 + 文件包含 → RCE
- IDOR + 越权 → 批量数据泄露
- 配置泄露 + 认证绕过 → 系统接管
- 低危 XSS + 高权限操作 → 管理员会话劫持
- SSRF + 云元数据 → 凭证窃取
- SQL 注入(有限) + 信息泄露 → 完整数据提取
- Prompt 注入 + Agent Tool(exec/文件) → 间接 RCE
- IDOR(读) + Mass Assignment(写) → 篡改他人数据

若组合成立，**在 chat 中单独输出一条组合链 finding**，标注关联的原始 finding 编号，独立定级。

**默认**：在验证汇总中增加 **「潜在组合（待确认）」** 小节，列出可能两两推高风险的 finding 对及简短理由；**不**对每条 finding 主动追问用户。若用户说 **「深度组合分析」**，再对高关联条目补充组合可行性说明。

### 二、验证结果汇总（在 chat 中直接输出）

全部验证完毕后，**在 chat 窗口** 直接输出汇总，不生成报告文件：

```
