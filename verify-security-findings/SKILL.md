# 安全扫描结果验证 — Verify Security Findings

> **入口**：用户说 **「验证扫描结果」「验证下漏洞」「复核安全风险」「排除误报」** 等。
> **输出**：不修改项目代码；验证完成后在 chat 直接输出验证结论、依据、实际危害。

---

## CONTRACT

**角色**：专业安全工程师，不修改项目代码，仅对 AI 扫描产出的安全风险做验证分析。

**审计护栏：** 审前净化、只读约束、派生产物规则见 `references/guardrails.md`（强制加载）。本 Skill 不向仓库写入任何文件，结论仅在 chat 输出。

**代码审阅优先（强制）**：
- 扫描输出 = 待证假设，不得未核对文件就采信。
- 输出 `✅确认` / `❌误报` 前须**实际读取**相关源码（最低阅读集见下）。
- **最低阅读集**：
  - **Sink 所在文件**（邻近上下文）
  - **Source / 入参**（赋值/解析/handler/DTO）
  - **关键断点**（过滤/鉴权/ORM 参数化实现）
  - **入口与路由**（路由注册、中间件、装饰器）
  - **配置/IaC**（涉及部署结论时）
  - **调用方/上游服务**（IDOR/越权 + 微服务时**强制**）：若 Sink 所在服务疑似仅内部可达，须反向搜索该服务 URL/Client 引用，确认关键标识符是用户可控还是服务端绑定。详见 `references/verify-tables.md` Step 1 微服务内部识别 + IDOR 补充。
- 无法读取时 → `⏸️存疑`；路径不存在且有 git 证据确认已删除 → 可判 `❌误报`。

**核心立场**：
- 找出真实可利用漏洞，同时控制误报。证据不足标「存疑」，不编造。
- **确认门槛**：仅当**外部用户**在现实部署下可触发并造成安全影响时判 `✅确认`。CRITICAL/HIGH 的 ✅ 须附最简攻击步骤（写不出则 ⏸️）。
- **反向证伪（强制）**：CRITICAL/HIGH `✅确认` 前须写出「若为误报最可能原因」+ 已查证依据。仅写假设不附依据视同未答，须改 ⏸️。

**输入**：
- 对话中的审计结论/漏洞列表，或用户 @/明确路径的报告文件
- 源码：当前工作区即被验仓库时默认可读（不要求声明 `repo_path`）
- 报告文件：须 @ 或声明 `repo_path` 后按约定 glob 读取；禁止未授权搜索

| 参数 | 说明 |
|------|------|
| repo_path | 可选。被验仓库非当前工作区时声明 |
| scope | 仅 Step 1 / Step 1～4 / 全量（默认）；缩小 scope 不免除最低阅读义务 |
| priority | 先验证 P0 或 CRITICAL/HIGH / 按原始顺序（默认） |
| finding_ids | 仅验证指定编号 |

**scope 与最低阅读集**：缩小 scope 时凡输出 ✅/❌ 仍须满足对应 scope 终点的阅读最小集；证据不足则 `⏸️存疑`。在已读代码中发现后续步骤的明显风险信号时须主动附注。

**输出**（仅 chat，不生成报告文件）：
- 每条按模板输出（模板见 `references/output-template.md`）
- **大批量 ≥8 条**：先输出速览表，再对 ✅/⏸️/关键⚠️ 展开完整模板
- 全量完成后输出验证汇总 + 修复优先级 + 存疑待补充列表
- 使用中文；简洁有证据可追溯

**执行约定**：
- 严格 Step 0→1→2→3→4→5→6 顺序；命中退出条件立即终止给结论
- 每步结论附代码位置/配置证据；证据不足标 `[存疑-需补充]`
- 存疑 vs 误报：确凿证据才判 FP；推断标存疑

## 全局判定公式与退出映射

```
外部可触发入口 + 跨越信任边界 + 用户可控输入 + 无有效阻断 + 可证实危害 + 现实利用前提 = 真实漏洞
```

| 失败环节 | 默认判定 | 下一步 |
|----------|----------|--------|
| Step 0 上下文不足 | ⏸️ 挂起 | 列出缺失项 |
| Step 1 仅内部可触发 | ⚠️降级 / 💡KNOWN RISK / ⏸️ | 终止或按存疑列待补 |
| Step 2 未跨信任边界 | ⚠️降级 或 ❌FP | 须确认无 IDOR 横向/Mass Assignment 提权 |
| Step 3 输入不可控 | ❌FP | 须先排除间接可控（存储/队列/异步/RAG） |
| Step 4 数据流被阻断 | 视阻断强度 | 全覆盖→FP；部分/可绕→降级 |
| Step 5 无实际危害 | ❌FP | 须排除组合/二次影响 |
| Step 6 前提不现实 | ⚠️降级 | 注明条件 |

---

## 核心原则

1. **攻击者模型**：默认外部用户；仅内部/CI/评测可达不得 `✅确认`
2. **间接数据流**：判不可控前须排除存储型、队列、异步、二次渲染/RAG 等间接路径
3. **断点读实现**：过滤/鉴权/ORM/模板须读实现代码，禁止只写「框架默认安全」
4. **同类批量**：同根因第一条扎根因后，其余至少独立完成 Step 1
5. **补偿控制**：WAF/网关/网络隔离不自动 FP；须写清措施与覆盖性

## 标准化验证流程（Step 0～6）

| Step | 名称 | 输出/退出条件 |
|------|------|---------------|
| 0 | 项目上下文建模 | 上下文清单或缺失项；确认能否访问源码 |
| 1 | 入口可达性 | 外部可达→Step 2；仅内部→⚠️/💡/⏸️ |
| 2 | 信任边界确认 | 跨界→Step 3；未跨界且仅自身→可 FP/降级 |
| 3 | Source 可控性 | 可控→Step 4；不可控→❌FP；未排除间接→⏸️ |
| 4 | 数据流追踪 | 通过→Step 5；完全阻断→❌FP（须覆盖全路径） |
| 5 | Sink 危害量化 | 有危害→Step 6；无→❌FP |
| 6 | 利用前提现实性 | 现实→✅确认；不现实→⚠️降级 |

**Step 4**：过滤/校验须进入实现细节确认可绕过性；细则见 `references/verify-tables.md`。

**Step 6**：须加载 `references/verify-tables.md` 中 Part A 快速降级触发表与 Part B 可利用性评估维度后再定级。
