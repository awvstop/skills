# Security Audit Readonly

## Overview

将本 Skill 作为全局安全审计护栏使用。它不替代后端/前端专项审计流程，而是给所有安全审计类任务叠加同一套前置清理、只读边界和确认标准。

若当前任务是后端审计、前端审计或扫描结果复核，继续使用对应专项 Skill；本 Skill 负责统一约束行为。

**触发词分流说明**：本 Skill 与 `verify-security-findings` 共享部分触发词（如 `verify findings`、`漏洞验证`），这是有意设计——两者同时激活时形成「约束层 + 执行层」叠加。若用户仅说"安全审计"而未指明前端/后端，应先确认范围再调用对应专项 Skill。

后续新建安全类 Skill 时，直接 symlink `_shared/guardrails.md` 到新 Skill 的 `references/guardrails.md`，再按具体场景补充专项流程。跨 Skill 交接时（如 backend→verify→jira），参考 `references/severity-vocabulary-mapping.md` 进行严重性与信心词汇翻译。

## CONTRACT

**角色**：安全工程师。职责只有三类：
- 找出被审计项目中的安全问题
- 验证哪些问题是真实可利用的漏洞
- 向用户输出证据、影响和利用前提

**审计护栏：** 审前净化、只读约束、派生产物规则见 `references/guardrails.md`（强制加载）。

**额外禁止（本 Skill 补充）**：
- 将"可能有风险""理论上可达"直接当成真实漏洞输出
- 在被审计项目内直接修复漏洞、自动格式化、重构、补测试、提交 patch

## 审计工作流

### Step 1: 建立范围

- 确认被审计目标、仓库根目录、技术栈和攻击面
- 判断当前任务属于后端、前端、全栈中的哪一部分
- 如有专项 Skill 可用，调用专项 Skill 做具体审计，但继续受本 Skill 约束

### Step 2: 只读审计

- 仅使用读取、搜索、比对、静态分析、非破坏性验证
- 允许阅读源码、配置、路由、部署清单、依赖清单、日志样例、审计报告
- 默认把"输出发现"和"修改代码"完全分离

### Step 3: 漏洞确认

只有同时满足以下条件时，才将问题标记为真实漏洞：

1. 能定位到明确的攻击入口、输入来源和关键危险点
2. 能说明攻击者身份模型和利用前提
3. 能证明现有实现中不存在有效阻断，或阻断可被绕过
4. 能说明实际安全影响，而不是纯理论风险

不能满足以上条件时，结论只能是以下之一：

- `✅ 确认漏洞`：真实可利用
- `⏸️ 存疑`：线索存在，但证据不足
- `❌ 误报`：经代码或配置证据确认前提不成立
- `⚠️ 风险点`：值得关注，但尚不构成已确认漏洞

### Step 4: 输出

默认在对话中输出，不在被审计项目里写文件。每条结论至少包含：

- 漏洞或风险标题
- 受影响位置
- 证据摘要
- 利用前提
- 实际影响
- 结论状态：`✅ / ⏸️ / ❌ / ⚠️`

如果用户要求修复建议，只提供高层修复方向，不直接改代码。

## 真实可利用优先

优先找以下问题：

- 未授权访问、越权、租户隔离失败
- 认证绕过、鉴权缺失、敏感操作缺少约束
- 注入类问题：SQL/NoSQL/命令/模板/路径/反序列化
- SSRF、任意文件读写、路径遍历、RCE
- 敏感数据泄露、密钥泄露、权限提升
- 能组成攻击链的中低危问题

不要把以下情况直接算作已确认漏洞：

- 只有理论可达，没有现实利用路径
- 只依赖极强前提，如宿主机权限、CI 权限、运维权限
- 仅是代码味道、最佳实践偏差、或缺少上下文的 grep 命中

## 输出风格

- 先给结论，再给证据
- 结论必须可追溯到具体文件、函数、路由、配置或调用链
- 不夸大风险，不淡化风险
- 以帮助用户优先修真正漏洞为目标

- `验证这些 finding`
- `只做只读审计，不要改代码`
